# はとナビ（鳩ナビ）おつかいクエスト — プロダクト詳細レポート

**バージョン:** 整合版（2026-06-21）／**対象ブランチ:** `feat/policy-alignment`
**位置づけ統一文:** 鳩ナビは、ピピットセルフを置き換えるものではなく、将来的にピピットセルフの買い物体験と接続可能な親子向け体験プロトタイプです。現時点では独立した Flutter Web デモとして動作し、将来的にはピピットセルフアプリ内統合、店舗貸出端末への搭載、または親のスマホでの買い物前後の補助利用を検討します。

> 本書は「気持ち悪いほど詳細」に、プロダクトの全容（思想・画面・データ・AI・ナビ・安全・技術・運用・将来）を1冊で説明する。実装の唯一の正は `docs/仕様書.md`、本書はその読み物版。

---

## 0. エグゼクティブサマリ
スーパーでの親子の買い物を「知育エンタメ」に変える Flutter Web アプリ。子どもは安全に売り場を巡り、商品をスキャンして食育クイズに答え、ポイント・バッジ・シールを集める。**買い物・会計の基盤はピピットセルフ（平和堂のスマホスキャン型セルフレジ）が担い、鳩ナビはそこに「学習性・安全性・ゲーム性」を上乗せする補助システム**。AIは「店内の通路ルート」を作らず、**選ばれた商品をどの順番で回るかを提案するだけ**。ナビ制御・方角・安全・進行はすべて端末内のローカルロジックで行い、AIが不安定でも検証とフォールバックで体験は壊れない。

---

## 1. 背景・課題・ターゲット
- **平和堂の経営課題**：少子高齢化の中、次世代ファミリー層（5〜8歳の子を持つ30〜40代）を競合から獲得しファン化する。
- **親の課題**：買い物中に子どもが退屈・ぐずる・走り回る。動画を見せる「受動的な時間」への罪悪感。
- **子の課題**：買い物に主体的に関われず退屈。
- **解決の核**：買い物を「能動的な知育・お手伝いの冒険」に変える＝親の罪悪感解消・時短、子の学びと達成感、店の来店動機・客単価維持・ロイヤリティ（三方よし）。

## 2. プロダクトの位置づけと、ピピットセルフとの関係
### 2.1 置き換えない（補助システム）
- **ピピットセルフの役割**：商品スキャン・商品登録・金額確認・専用精算機での会計（買い物と会計の基盤）。
- **鳩ナビの役割**：安全ナビ・お約束確認・逐次ミッション・食育クイズ・ポイント/バッジ・シール引換演出（学習・安全・ゲーム性）。
- **会計は鳩ナビに実装しない**。鳩ナビ内の「スキャン」は購入処理ではなく、**「みつけた！」というゲームの合図**（クイズのトリガー）。完了画面は「ピピットセルフでお会計してね」と案内するのみ。

### 2.2 現実的な3つの運用パターン
| パターン | 内容 | 位置づけ |
|---|---|---|
| **A（理想・将来構想）** | ピピットセルフアプリ内に子ども向けモードとして統合（入店〜スキャン〜会計＋おつかい体験を1つに）。要：アプリ改修・API連携・セキュリティ・店舗運用設計 | 将来 |
| **B（現実的）** | 店舗貸出端末に鳩ナビを搭載。親はピピットセルフで会計、子は鳩ナビ。2台運用で衝突しない | 実運用候補 |
| **C（デモ／前後補助）** | 親のスマホで鳩ナビを起動し、買い物前の商品選択・買い物中のミッション・買い物後のポイントに使う。親子で受け渡しながら一緒に。**両アプリ同時切替運用は非現実的** | 現デモに近い |

- **現時点**：独立した Flutter Web デモ。ピピットセルフとの直接連携は**未実装**（接続は将来構想として提示）。

## 3. 全体フロー
```
ホーム
 → 買い物リスト（親が8品から選ぶ）
   → [おつかいの順番をきめる：ローカル基本順 ＋ AI提案 ＋ 回遊性ガード]
     → お約束（はしらない/まえをみる/おとなのそばにいる・音声解放）
       → 移動：方位磁針ナビ（次の1売り場だけ）＋走行ロック＋危険アラート

 ※ 途中追加：移動画面の🛒アイコンから「リスト外」もスキャン→同じくクイズ→+1（条件分岐）
```

## 4. 画面ごとの詳細仕様
### 4-1 HomeScreen（`home_screen.dart`）
- StatelessWidget。はとっぴーアバター＋「クエストをはじめる！」→ ShoppingListScreen。

### 4-2 ShoppingListScreen（`shopping_list_screen.dart`）
- `sampleItems`（**8件**）をタップ選択（検索ボックスはダミー）。1件以上で「**おつかいの順番をきめる（N件）**」活性化。
- 押下で「おつかいの順番」演出（最低約1.2秒）→ 裏で巡回順を確定（§6）→ お約束画面へ。

### 4-3 SafetyPledgeScreen（`safety_pledge_screen.dart`）
- 親→子の受け渡し儀式。「はしらない／まえをみる／おとなのそばにいる」。
- 「やくそくした！スタート！」押下で **`SpeechService.unlock()`（iOS音声解放）** → NavigationScreen へ pushReplacement。

### 4-4 NavigationScreen（`navigation_screen.dart`）＋ CompassScreen（`compass_screen.dart`）
- 次の**1売り場だけ**を提示（全ルート非表示＝走り出し防止）。進捗バー＋ミッション番号。
- 移動UIは CompassScreen を埋め込み（ミッションごとに `ValueKey('compass-<id>')` で再生成）。
- **方位センサー**で方角（円環スムージング＋`baselineOffsetDeg`＋最短回り）。**絶対方位優先・相対と混在させない**（象限ズレ対策）。
- **走行ロック**：加速度センサーで「子の平均歩行速度超え（早歩き/走り）」を0.5秒持続で検知→赤い全面「とまって！」＋**音声**。非対応端末は出さない（壊れない）。
- **危険アラート**：2件目到達で赤ダイアログ＋**音声**。
- AppBar の🛒で**途中追加**（リスト外スキャン）。到着ボタンで QuizScreen。

### 4-5 QuizScreen（`quiz_screen.dart`）
- フェーズ：idle → camera（実カメラ `mobile_scanner`）→ adding（カゴ追加演出・裏でAIクイズ生成）→ quiz（4択）→ result。
- スキャン＝「みつけた！」の合図。正解で `pop(true)` ＋限定バッジ。カメラ不可は「つぎへ」でフォールバック。

### 4-6 StickerScreen（`sticker_screen.dart`）
- 完了演出「**ピピットセルフでお会計してね**」（会計はしない）→ シール引換券（疑似QR）→ バッジ図鑑。
- `initState` で `PointService.add(earnedPoints)` を1回。→ PointScreen へ。

### 4-7 PointScreen（`point_screen.dart`）
- 今回P＋累計P（`PointService.loadTotal`）。累計≥10でシール交換バナー。ホームへ。

## 5. データモデル（`lib/models.dart`）
### 5.1 ShoppingItem
`id / name / areaId / area(表示名) / emoji / question / choices[4] / correctIndex / explanation / badgeName / badgeEmoji / janCode?`
- `areaId` は売り場マスタ `storeAreas` のキー。巡回順・逐次ナビはこの areaId 基準。

### 5.2 売り場マスタ `storeAreas`（`StoreArea{id,label,pathIndex,x,y}`）
- `売り場マップ.pdf` を**開発時に1回デジタル化**した固定データ。25売り場を `pathIndex`（入口→…→レジの一方向スイープ順）付きで保持。x,yは方角強化用（現在0.0）。
- **実行時に画像は読まない／AIに画像を渡さない**。棚替え時のみ更新（運用コスト最小）。

### 5.3 おつかい商品（8件・架空ブランド＋一般名）
| id | 表示名 | areaId / 売り場 |
|---|---|---|
| vegetables | みずうみ農園のトマト | local_vegetables / 地場野菜 |
| milk | あおぞら牧場のぎゅうにゅう | dairy / 乳製品 |
| egg | やまびこ農園のたまご | egg / 卵 |
| yogurt | あおぞら牧場のヨーグルト | yogurt / ヨーグルト |
| bread | こむぎ工房のパン | bread / パン |
| meat | みのり牧場のおにく | meat / お肉 |
| fish | みなと水産のおさかな | fish / お魚 |
| fruits | ひだまり農園のくだもの | fruits / 果物 |
- **実在商品名は使わない**（架空ブランドでリアル感を担保）。クイズは「地元のものは新鮮・農家/牧場の応援になる」の食育トーンを一般表現で。
- `bonusItem`（途中追加用・areaId=`unknown`）：商品名が分からなくても出せる一般食育クイズ＋「はっけんはとっぴー」。

## 6. 巡回順ロジック（AI＝順番提案だけ／ナビ＝ローカル）
1. **ローカル基本順**：選択商品を `storeAreas[areaId].pathIndex` 昇順でソート＝**定義上後戻りしない**（回遊性を保証）。AI不要で必ず成立。
2. **AI提案**：`GeminiService.suggestVisitOrder(items)`（入力 `{id,name,areaId}` → 出力 `{"order":[id]}`）。検証＝全ID含む/未知ID無/重複無/件数一致。
3. **回遊性ガード**：AI順の「後戻り回数（pathIndexが前より小さくなる回数）」がローカル順以下のときだけ採用。悪ければローカル順。失敗/キー無も**ローカル順へフォールバック**。
4. **呼び出しは開始時の1回のみ**。ナビ中はGeminiを呼ばない。現在地＝直前にスキャンした売り場、方角＝方位センサー＝すべてローカル。
- 行って戻る（両端の品・途中追加）が残っても、1個ずつ矢印表示なので体験は壊れない（寄り道扱い）。

---

## 6.5 方角ナビ（針が「次の売り場」を指す）
- `storeAreas` に各売り場の近似座標 `x,y`（マップ準拠）を保持。NavigationScreen が **現在地（直前にスキャンした売り場、最初は `start`）→ 次の売り場** の方位角 `targetBearingDeg = atan2(dx, dy)`（北=0・時計回り）を計算し、CompassScreen に渡す。
- CompassScreen の針は `needle = targetBearingDeg + storeNorthOffsetDeg − heading` を最短回りで回転＝**スマホをどの向きに持っても次の売り場の方向を指す**。座標が無い場合は従来の方位（北指し）にフォールバック。
- **ワンタップ較正「むきを あわせる」**：実際に売り場の方を向いて押すと `storeNorthOffsetDeg = heading − targetBearingDeg` を設定し、店舗マップ北と地磁気北のズレを吸収（全ミッションで保持）。これにより測位なしでも実用的な精度に。
- 精度：円環スムージング＋単一ソース（絶対方位優先）＋最短回り＋較正。GPS/BLEは不使用。

## 7. AIの活用（場所・内容・制御）
| 場所 | 内容 | 入力 → 出力 | タイミング | 失敗時 |
|---|---|---|---|---|
| 買い物リスト | 巡回順の提案（順番だけ） | `{id,name,areaId}` → `{order:[id]}` | 開始時1回 | ローカル順 |
| クイズ | 食育クイズ生成 | 商品名・売り場 → `{question,choices[4],correctIndex,explanation}` | 商品ごと1回 | 固定クイズ |
- 経由：`api/gemini.js`（Vercel Function）→ **gemini-2.5-flash**。APIキーはサーバ側に秘匿。
- **ハルシネーション対策**：出力検証＋固定値フォールバック（デモは止まらない）。
- **AIがやらないこと**：現在地推定／通路ルート生成／方角計算／危険判定／画面遷移／スキャン処理／精算。
- **年齢別クイズレベル（実装済み）**：ホーム画面で「おこさまの がくねん（クイズのむずかしさ）」を3段階（ちいさめ/ふつう/むずかしめ）で選択（`LevelService`・shared_preferences で保持）。`generateQuiz(item, level)` が level・levelHint を `api/gemini.js` に渡し、難易度・言葉づかいを調整。失敗時は固定クイズへフォールバック。
- **さらなる教育エンジン化（将来）**：正誤履歴による適応難易度・会話/解説生成・「きょうのまなび」サマリ。呼ぶのは開始/商品ごと/完了のみ（移動中は呼ばない）。

## 11. デモ範囲と将来構想（明確に分離）
- **現時点（デモ）**：独立Webデモ。巡回順提案・クイズ生成は実際に Gemini で稼働（検証＋フォールバック）。ピピットセルフ直接連携・会計は未実装（演出）。
- **将来**：ピピットセルフ内統合（理想）／店舗貸出端末搭載（現実的）。AIの教育エンジン化（適応学習・学びサマリ）。データ資産化（知育×購買）でLTV・差別化。

## 12. 中間評価（SoftBank AI 6ペルソナ）への対応
- **AI活用度は強み**（≈4.0）。弱点は「構想止まりで動くデモがない」→ Geminiを**実稼働**させて解消。ハルシネーションは検証/フォールバックで回答。安全・端末オペ・収益/スケールは本実装＋発表で補強。

## 13. 既知の課題
