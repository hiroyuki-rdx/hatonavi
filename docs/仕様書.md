## 0. プロダクトの位置づけ（戦略）

- 平和堂×ソフトバンク×龍谷大学のハッカソン企画。親子の買い物を知育エンタメ化する Flutter Web アプリ。
- **位置づけ統一文**：鳩ナビは、ピピットセルフを置き換えるものではなく、将来的にピピットセルフの買い物体験と接続可能な親子向け体験プロトタイプです。現時点では独立したWebデモとして動作し、将来的にはピピットセルフアプリ内統合、店舗貸出端末への搭載、または親のスマホでの買い物前後の補助利用を検討します。
- **想定する3つの運用パターン**：
  - **A** ＝ ピピットセルフアプリ内に子ども向けモードとして統合（理想・将来構想）。
  - **B** ＝ 店舗貸出端末に搭載（現実的。親はピピットセルフで会計、子は鳩ナビ）。
  - **C** ＝ 親のスマホで買い物前後の補助（現デモに近い。両アプリ同時切替運用は非現実的）。
- **AI が行うこと**：① 巡回順の提案（回る順番だけ）／② 食育クイズ生成（将来：適応難易度・会話/解説・学びサマリ）。
- **AI が行わないこと**：現在地推定／通路ルート生成／方角計算／危険判定／画面遷移／スキャン処理／精算。ナビ制御はローカル。
- **会計**：鳩ナビは会計・商品登録・精算を実装しない（ピピットセルフの役割）。鳩ナビのスキャンは「みつけた！」というゲームの合図であり、決済処理ではない。
- 巡回順の提案＝ Gemini、方角＝方位センサー（コンパス）、進行確定＝スキャン、の3点で**測位ハードウェア（BLE/GPS）不要**。スキャンは「みつけた！」の合図（会計ではない）。

## 1. 画面仕様一覧

### 画面遷移マップ（実装準拠）

```
HomeScreen
  └─→ ShoppingListScreen（買い物リスト）
        └─→ [おつかいの順番ダイアログ（巡回順提案・擬似1.6秒）]
              └─→ SafetyPledgeScreen（お約束）
                    └─→ NavigationScreen（逐次ミッション・本体）
                          │   ├ 内部に CompassScreen を埋め込み（方位センサーで方角表示）
                          │   ├ 2件目到達時に hazardAlert ダイアログ
                          │   └ 「ここに とうちゃく！」→ QuizScreen
                          └─→ QuizScreen（スキャン演出→クイズ→結果）
                                └─→ 次ミッション or StickerScreen（全完了）
StickerScreen（決済演出→シール引換券→図鑑）
  └─→ PointScreen（獲得P＋累計P・シール交換判定）
        └─→ HomeScreen（pushAndRemoveUntil）
```

### CompassService（`lib/services/compass_service.dart`）実装済・Web専用
- `Stream<double> get headingStream`：`deviceorientationabsolute`（無ければ`deviceorientation`）を購読。**iOSは `webkitCompassHeading`、Android等は `alpha`→`360-alpha`** で北基準0〜360度に正規化。

- 実装は `dart:html` + `dart:js_util`（Web専用）。ネイティブ化時は `motion`/`sensors_plus` 等へ差し替え。

## 7. AIによる巡回順提案＋ローカル逐次ナビ（Q6回答・BLEなし）

AI は「回る順番」だけを提案し、現在地推定・方角計算・画面遷移などのナビ制御はすべてローカルで行う。

### 7-1. 店内ナビのローカル制御
1. **巡回順**＝Gemini が買い物リストの売り場（areaId）を**回る順番だけ**提案する（物理ルート・方角は生成しない）。失敗時はローカルの固定順にフォールバック（Phase 3）。
2. **現在地**＝GPS/BLE 不使用。連続測位しない。**「直前にスキャンした売り場」を現在地とみなす**。スタートは `start` 売り場で確定。
3. **方角**＝静的売り場マスタ上の現在→次エリアの相対方位を、方位センサー（コンパス）で実世界の向きに合わせ矢印表示（計算はローカル）。
4. **逐次提示**＝今向かう1個だけ表示し、完了ごとに次へ。全体の順番は見せない（走り出し防止）。
5. **校正**＝マップ座標と磁北の対応（店舗ごとの回転オフセット1個）。デモは北基準仮定/オフセットをハードコードで吸収。
→ BLE/GPS/Wi-Fi測位ゼロ。必要なのは静的売り場マスタ1枚と方位センサーのみ。

### 7-2. AI 巡回順 入出力スキーマ
- **入力**：`selectedItems`＝選択商品の配列 `[{ id, name, areaId }, ...]`（＋売り場マスタ）。画像は渡さない。
- **出力**：`{ "order": [id, ...] }` ＝ **回る順番（商品 id の並び）だけ**。物理ルートではない。

```json
// 入力例
{ "selectedItems": [
    { "id": "milk",   "name": "あおぞら牧場のぎゅうにゅう", "areaId": "dairy" },
    { "id": "fruits", "name": "ひだまり農園のくだもの",     "areaId": "fruits" }
] }
// 出力例
{ "order": ["fruits", "milk"] }
```

### 7-3. 出力検証・フォールバック・回遊性ガード
- **検証**：出力 `order` が「選択全 id を含む／未知 id を含まない／重複が無い／件数が一致する」の4条件をすべて満たすこと。
- **フォールバック**：検証 NG・API 失敗・`order` キー無 のいずれかの場合は、ローカルの **pathIndex 昇順**順に切り替える。
- **回遊性ガード**：AI 提案順が pathIndex 順より**後戻り（手前の pathIndex へ戻る動き）が大きい**場合は AI 順を破棄し、pathIndex 昇順へフォールバックする。

## 8. Gemini API 連携仕様（Phase 3・将来）

- モデル：**`gemini-2.5-flash`**（`gemini-2.0-flash` は廃止のため不可）。
- 用途：①**巡回順の提案**（売り場マスタ＋リスト→回る順番だけ。物理ルートは生成しない）②**クイズ生成**（商品名→4択クイズ。将来：適応難易度・会話/解説・学びサマリ）。
- 入出力・検証・フォールバック・回遊性ガードは「7-2／7-3」に準拠。**画像は渡さない**（売り場マスタはテキストの areaId のみ）。
- キー秘匿：**Vercel Serverless Function（`api/gemini.js`）でプロキシ**し `GEMINI_API_KEY` を環境変数で保持（フロントに出さない）。
- ハルシネーション対策：出力を既知 areaId/id・テンプレに制約＋事実照合、**失敗時は pathIndex 昇順・既存クイズにフォールバック**。
